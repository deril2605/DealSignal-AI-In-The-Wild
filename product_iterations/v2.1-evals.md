# DealSignal MVP v2.1-evals

## Working Title

Human evaluation loop for ranked opportunities.

## Status

Implemented as a first pass.

`v2.1-evals` now exists in code and UI. The remaining work is mainly operational use over multiple days and later tuning from the collected labels.

## Goal

Move from "we have a ranked top-10" to "we can validate whether the ranking is actually useful."

`v2` produces ranked opportunities.

`v2.1-evals` adds a persisted human-review loop on top of those rankings.

## Product Thesis

`v2` made prioritization possible, but it still relied on heuristic scoring.

The next step is not more ranking complexity. It is collecting structured human feedback on the ranked outputs so future tuning is grounded in real judgments.

## Desired User Outcome

A user should be able to:

- open a daily review queue
- see that day’s top ranked opportunities
- mark each as:
  - `useful`
  - `maybe`
  - `not useful`
- leave short notes
- come back later and still see older review sets

## What Shipped

### Data Model

New table/model:

- `opportunity_evals`

Stored fields include:

- `snapshot_date`
- `rank`
- `company_id`
- `signal_event_id`
- `lead_score_id`
- `lead_score_value`
- `review_status`
- `review_notes`
- `created_at`
- `reviewed_at`

### Pipeline

After the normal pipeline completes:

1. select the top opportunities for the last 7-day window
2. create a daily snapshot for the current IST date
3. store up to 10 eval rows
4. skip duplicate snapshot creation if that date already exists

This means rerunning the pipeline on the same day does not duplicate the daily review set.

### UI

New route:

- `/evals`

The page provides:

- grouped daily eval snapshots
- newest day expanded by default
- older days collapsible
- one-click actions:
  - `Mark Useful`
  - `Mark Maybe`
  - `Mark Not Useful`
- notes persistence
- historical visibility across dates

### Persistence

Review actions update SQLite and then immediately sync the DB back to Blob.

This avoids losing local review decisions on the next app restart.

## Current Review Semantics

- `useful`
  - worth a quick human look
- `maybe`
  - somewhat interesting but not clearly strong
- `not useful`
  - weak, noisy, generic, or not actionable

These labels are intentionally simple so that review friction stays low.

## Current Workflow

1. ACA or local pipeline run completes
2. top-10 daily eval snapshot is stored
3. user opens `/evals`
4. user reviews items now or later
5. feedback remains attached to the original snapshot date

This means review does not need to happen on the same day as the run.

## Why This Version Matters

This is the bridge between:

- heuristic ranking
and
- evidence-based tuning

Without this layer, `v2` can rank opportunities, but there is no durable validation dataset showing whether the ranking is actually useful.

## Success Metric

Primary:

- after multiple days of review, the system has a usable labeled history of top-ranked opportunities

Operational:

- daily review sets are reliably stored
- users can review them later without losing history
- review labels persist across restarts and cloud sync

## Known Limitations

- no automated tuning from eval labels yet
- one shared reviewer model only; no per-user reviewer identity
- no summary analytics page yet for useful-rate over time
- no export workflow yet for offline threshold/weight tuning

## Next Work After v2.1-evals

1. collect 1-3 weeks of eval labels
2. analyze useful / maybe / not useful distributions
3. tune `v2` scoring weights and thresholds from real feedback
4. add eval analytics in admin or a dedicated report view
5. optionally automate weight search from labeled history

## Non-Goals For v2.1-evals

- automatic model retraining
- CRM integration
- outbound review workflows in Slack/Teams/email
- reviewer identity and permissions

## Decision Standard

`v2.1-evals` is successful if:

- daily top-opportunity snapshots are preserved
- humans can label them later
- labels persist durably
- the project now has the data needed to tune ranking quality with evidence instead of intuition
