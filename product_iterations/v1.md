# DealSignal MVP v1

## Working Title

Change detection plus alerting.

## Status

Implemented as a first pass.

`v1` now exists in code and UI. The remaining work is mainly tuning alert quality rather than building the core system.

## Goal

Move from "collect and display all detected signals" to "surface only meaningful change."

The system now compares a company's last known narrative state against newly extracted event state, records the delta, and marks it as either:

- `ALERT`
- `RECORDED`

## Product Thesis

`v0` proved the pipeline could discover, extract, dedupe, score, and store signals.

`v1` adds a second layer:

- maintain company-level narrative state
- detect deltas against that state
- prioritize only the changes that appear materially important

## What Shipped

### Data Model

Two new tables/models were added:

- `company_narratives`
- `narrative_deltas`

These support:

- latest known company narrative snapshot
- per-event delta record
- significance score
- `should_alert` classification

### Pipeline

After extraction, each new event now goes through deterministic change evaluation:

1. load prior company narrative
2. build current event state
3. merge current state into company narrative
4. compute newly introduced values
5. score significance
6. mark the delta as `ALERT` or `RECORDED`
7. update the company narrative snapshot

### UI

`v1` is surfaced in the UI:

- home page prioritizes alerts first
- home page supports change-status filtering
- company page shows:
  - last known narrative
  - recent changes
  - event list with alert/recorded context
- event detail shows delta reasoning
- admin page shows delta counts

## Current Definition Of Change

The current implementation treats these as meaningful new additions when newly observed:

- geography
- vertical
- theme
- counterparty
- strategy language

These are compared in structured form rather than using freeform narrative-vs-narrative semantic comparison.

## Current Alert Logic

Each delta gets a significance score and is marked as `ALERT` only when it crosses a threshold.

Current characteristics:

- structured-first
- deterministic
- rule-based
- explainable in UI

This was chosen to keep `v1` debuggable and testable.

## Current User Outcome

A user can now answer:

- what changed for this company?
- was that change considered important?
- why did the system classify it as alert-worthy or not?

This is visible directly in the main feed, company pages, event pages, and admin page.

## What Is Still Heuristic

The following areas are intentionally first-pass and may still need refinement:

- vertical inference
- theme quality
- strategy phrase extraction
- significance weighting
- threshold sensitivity

These are tuning concerns, not architecture gaps.

## Known Limitations

- some labels may still feel generic or too broad
- some important changes may be marked `RECORDED` until scoring is tuned
- some weak changes may still be marked `ALERT`
- alert quality has not yet been validated over a large number of runs

## Success Metric

Primary:

- lower alert noise
- higher relevance than `v0`

Practical evaluation:

- when reviewing the feed, `ALERT` items should usually feel worth attention
- `RECORDED` items should feel like lower-priority tracking entries rather than missed alerts

## Non-Goals For v1

- email, Slack, or Teams delivery
- advanced semantic narrative comparison
- large-scale production hardening
- final tuning of alert quality

## Next Work After v1

Likely next steps:

1. review alert quality on real runs
2. tune significance weights and threshold
3. tighten vertical/theme/strategy inference
4. refine UI language and presentation
5. only later consider outbound alert delivery

## Decision Standard

`v1` is complete enough if:

- the system stores and surfaces narrative deltas
- alerts are visually prioritized
- users can inspect why a change was flagged
- the implementation is stable enough for real-run tuning
