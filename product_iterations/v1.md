# DealSignal MVP v1

## Working Title

Change detection plus alerting.

## Goal

Move from "collect and display all detected signals" to "surface only meaningful change."

The core idea is to compare what is newly observed for a company against what is already known, then alert only when there is a material delta.

## Product Thesis

`v0` proves that the system can discover, extract, dedupe, score, and store signals.

`v1` should make the output more useful by reducing noise:

- fewer repetitive alerts
- stronger signal-to-noise ratio
- better indication of strategic movement, not just article volume

## Target User Outcome

A user should be able to look at the system and quickly answer:

- what materially changed for this company since the last known state?
- why does this matter?
- is this genuinely new or just another article saying the same thing?

## Proposed Definition Of Change

Alert when one or more of the following are newly observed or meaningfully changed:

- new geography
- new industry or customer vertical
- new strategic theme
- new counterparty or partner
- new product or market expansion language
- stronger commitment language than before
- change in timeline or magnitude
- repeated confirmation from multiple independent sources

Do not alert on:

- duplicate coverage
- weak rewording of known strategy
- low-information PR language
- generic statements without a new fact

## Proposed v1 Output

Instead of treating every extracted event as equally alert-worthy, produce:

- regular stored signal events
- a company-level "narrative snapshot"
- a delta record explaining what changed
- an alert decision with reason

This means the system can still store all events, but only elevate the ones that meaningfully move the company's narrative.

## Proposed Data Model Additions

### `company_narratives`

One row per company representing the latest known strategic narrative snapshot.

Suggested fields:

- `company_id`
- `summary`
- `geographies`
- `verticals`
- `themes`
- `counterparties`
- `strategic_phrases`
- `latest_event_id`
- `updated_at`

### `narrative_deltas`

One row per evaluated change event.

Suggested fields:

- `company_id`
- `source_event_id`
- `previous_narrative_id`
- `new_narrative_id`
- `delta_type`
- `delta_payload`
- `significance_score`
- `should_alert`
- `reason`
- `created_at`

## Proposed Pipeline Extension

After extraction and scoring:

1. load current company narrative snapshot
2. derive structured narrative inputs from the new event
3. compare new event state with prior known state
4. compute delta categories
5. score significance of the delta
6. store the delta decision
7. update company narrative snapshot
8. surface only alert-worthy deltas in admin/UI or future outbound channels

## Recommended Implementation Strategy

Use a structured-first diff, not a freeform narrative-vs-narrative LLM comparison.

Why:

- easier to debug
- lower cost
- less drift
- deterministic enough for unit tests
- easier to explain to users

Recommended first-pass logic:

- compare normalized arrays such as geography, counterparties, themes
- compare key phrases extracted from event summaries
- use a rule-based significance model first
- optionally use an LLM later only to summarize why the change matters

## First-Pass Significance Heuristic

Example scoring approach:

- new geography: high
- new vertical: high
- new counterparty: medium to high
- new strategic theme: medium
- stronger wording only: low
- repeated confirmation from another source: score boost
- low-confidence event: score penalty

Alert if significance crosses threshold.

## UI Changes For v1

Potential additions:

- "New Changes" feed on home page
- company detail section showing last known narrative
- admin page metric for total alerts vs total events
- badges showing delta categories such as `New Geography`, `New Vertical`, `New Partner`

## Success Metric

Primary:

- lower alert noise
- higher proportion of alerts judged relevant by a human

Secondary:

- fewer duplicate or restatement events shown as important
- easier review workflow for a user following a company

## Risks

- too much reliance on LLM semantic comparison will be hard to trust
- weak structured extraction will produce noisy diffs
- over-alerting if significance rules are too loose
- under-alerting if rules are too conservative

## Suggested Implementation Order

1. add narrative snapshot and delta tables
2. implement deterministic diff logic
3. add significance scoring and `should_alert`
4. expose delta info in UI/admin
5. evaluate output quality on real watchlist runs
6. only then consider outbound alert delivery

## Non-Goals For v1

- email, Slack, or Teams delivery
- fully autonomous reasoning about strategy
- perfect semantic understanding of every article
- large-scale production hardening

## Decision Standard

`v1` is successful if a user sees fewer but more meaningful surfaced updates than in `v0`, and can inspect exactly why the system considered a change material.
